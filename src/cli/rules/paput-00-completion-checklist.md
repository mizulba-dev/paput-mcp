## PaPut Completion Checklist

Run this before writing the final message of any turn in which work was done — a task finished, a problem solved, a design decision settled, memos or notes created, or a repeatable procedure established. Run BOTH checks below, in order, every time. They are independent: one finding nothing says nothing about the other, and doing one is never a substitute for the other.

Skill naming: these rules refer to PaPut skills by their standalone names (`paput-capture`, `paput-save`, `paput-harvest`, `paput-project-document`, ...). When PaPut is installed as a plugin instead, the same skills live under the `paput` namespace — read `paput-<name>` as `paput:<name>` (e.g. `paput-capture` → `/paput:capture`). Every rule below applies to whichever form is installed.

- [ ] Check 1 — Project document. Did this work settle a project-specific design decision (`design_doc`) or complete a repeatable project procedure (`procedure`)? If yes, save it with `paput_add_project_document` and report what you saved. If no, say "Check 1: no project document needed" so the check is visibly done.
- [ ] Check 2 — Reusable knowledge. Did reusable cross-project knowledge appear? If yes, follow the `paput-capture` workflow and report the title, categories, and candidate ID. If no, say "Check 2: no reusable knowledge found" so the check is visibly done.
- [ ] Session marker — a conditional follow-up, not a third check. If no capture registered candidates this session (Check 2 ended "none found", or a capture attempt registered nothing — only actual registration marks the session; Check 1 alone does not mark it either, so run the marker in those cases too) and the session is not marked yet, mark the current session as processed for harvest with `paput_mark_processed_session`, resolving the real session identity as in the `paput-capture` skill's Source Session Identity section (Claude: the session's 36-character UUID; Codex: the rollout basename located from `CODEX_THREAD_ID`). Once per session is enough — skip when an earlier turn already marked it or a capture call actually registered candidates (registration marks as a side effect). Skip entirely when operating as a spawned agent (a subagent, teammate, or `codex exec` run — recognizable from your spawn context): the parent session runs its own checklist, and agent-driven transcripts must stay marker-free (`paput-harvest` filters them by origin instead). This keeps light sessions (quick questions, small investigations) from piling up as unprocessed backlog in `paput-harvest`. Marking the running session is intended: if knowledge appears in a later turn, capture simply re-marks it. If the marker call fails, note it briefly and move on — never block the reply on it. Mention it only when it acted (e.g. "session marked"); when skipped, staying silent is fine.

State the one-line outcome of EACH check. Never leave a check silent. The detailed procedure for each check is in the sections below, and the self-check at the end of this block confirms both were run.
